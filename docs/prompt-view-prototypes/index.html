Per-Shot Prompts View — three prototypes

Backlot · phase E1.9 · prototype gallery
Each prototype shows the same data: Shot 01 — Wide establishing with 6 prompt variations across 3 types and 4 statuses. Click into each to feel how it reads. The decision shapes E1.9 onward — same underlying filesystem (scenes/.../shots/<id>/prompts/v*.md) regardless of which UI we pick, so this is purely a UX call.
Option A · safest

Card grid

Moodboard-style. Every variation visible at once, dense, filterable by type / status. Click a card to focus, drag to reorder. Familiar pattern (Pinterest, Linear).

Best for: at-a-glance overview, batch decisions
Build cost: low (~2 days)
Trade-off: scrolls a lot at 12+ prompts; less visual hierarchy

Option B · most ergonomic for editing

Split view

List of variations on the left, full editor on the right. One-click switch between prompts. Best for editing one prompt at a time with iteration tree + version history visible per prompt.

Best for: deep work on one prompt at a time
Build cost: low (~2 days)
Trade-off: can't compare two prompts side-by-side
Open Option B →
Option C · highest ceiling

Canvas

Free-positioned cards with iteration arrows. The "tree of exploration" made visible. Pan / zoom, drag-drop relationships, visual lineage at a glance. Killer demo, more code.

Best for: visual thinkers, branchy iteration trees
Build cost: medium (~1-1.5 weeks, needs React Flow + Dagre)
Trade-off: overkill for < 8 prompts; learning curve
Open Option C →

My recommendation, after building all three

Ship Option A (card grid) as v1 — it covers the observability + iteration goals at the lowest cost. The data model (file per prompt, frontmatter linking parent → child, git history) is identical across all three, so we can layer Option C (canvas) on later as a "see all variants" mode for power users without re-architecting. Option B is the worst of both worlds — single-prompt focus loses the moodboard vibe, and it's not as visual as either A or C.
