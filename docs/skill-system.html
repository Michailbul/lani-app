Skill system —
build spec

Decisions locked. This is what gets built: how skills are stored, shipped, chosen on the Settings page, passed to the agent, and created by the agent.

The model. One directory — ~/.lani/skills/ — holds every skill the Lani agent can use, one folder each. ~/.lani/ is registered with the SDK as a local plugin, so skill discovery is independent of settingSources. The user controls which skills are active in Settings; the agent creates and edits skills through a reviewed diff. The user's real library at ~/.claude/skills is never read directly by the agent.

Imported and agent-created skills are one file reachable from both directories — edits stay consistent everywhere. Factory skills are Lani-local copies (they have no ~/.claude origin). Reviewed editing is what makes the shared-file model safe: no change lands without your approval.

  options: {
    cwd: "<the project worktree>",          // what the agent edits

    skills: disabledEmpty ? "all" : enabledNames,  // lani:<slug>
    settingSources: loadProjectClaudeMd ? ["project"] : [],

                    append: laniHarness },
  }
})
  • Discovery — the plugins option loads every skill in ~/.lani/skills/. Independent of settingSources.
  • Enabling — skills carries the active set (all minus the disabled list). Plugin skills are namespaced lani:<slug>; the UI shows plain names.
  • CLAUDE.md — settingSources only ever holds "project" (toggle on) or nothing (off). "user" is never used — that closes the ~/.claude/CLAUDE.md leak.
C

The Settings page — controlling inclusion

A Skills entry in Settings opens the page. It has three parts.

1 · Active skills

A flat, alphabetical list of every skill currently in ~/.lani/skills/ — factory, imported, and agent-created together — each with an on/off toggle.

  • Toggle off → the slug is written to ~/.lani/skills-disabled.json. The agent stops seeing it next session.
  • Toggle on → slug removed from that file.
  • Anything newly added — seeded, imported, or created — is on by default (it's simply not in the disabled list).
  • A Remove action per skill: an imported skill drops just its symlink (your real ~/.claude skill survives); a factory or created skill removes the folder.

2 · Import from your library

This is the flag/section you described. It scans the union of ~/.claude/skills + ~/.agents/skills (deduped by slug), and lists skills not yet in Lani.

  • Add all — symlinks every listed skill into ~/.lani/skills/ at once.
  • Add per skill — pick individually.
  • "Add" = create a symlink ~/.lani/skills/<slug> → the user's skill. Instant, no file copying. Added skills appear in the Active list, on by default.

3 · Behaviour toggles

  • Load project CLAUDE.md — default ON. On → settingSources:["project"]; off → [].
  • Publish agent-created skills to ~/.claude/skills — default ON. On → a new skill the agent creates is symlinked into your library so other tools see it.
D

How the agent creates & edits skills

Every skill change the agent makes is reviewed. The agent never writes a SKILL.md with raw file tools — the skills-gate blocks that and redirects it to propose_skill_change.

  1. The agent drafts the skill (guided by the adapted skill-creator).
  2. It calls propose_skill_change → you see a diff modal.
    • Editing an existing skill: a normal old-vs-new diff.
    • Creating a new skill: the full proposed SKILL.md shown as an all-new diff to approve.
  3. You Approve or Dismiss.
  4. On Approve, Lani writes the skill into ~/.lani/skills/<slug>/.
  5. For a new skill, if "Publish to ~/.claude/skills" is on and the name is free there, Lani also drops a symlink ~/.claude/skills/<slug> → the new folder.

Delete — the agent asks; you get a plain confirm (no diff — nothing to diff). On confirm the folder is removed; if it had a ~/.claude symlink, that's removed too.

Why reviewed editing makes the shared file safe Imported and created skills are one file shared with ~/.claude. Because every change is a diff you approve first, "consistent everywhere" never means "silently broken everywhere." The review gate is the safety that lets the symlink model work.
E

Harness & skill-creator changes

Yes — both need updating. The stock skill-creator skill is generic; it doesn't know ~/.lani/ exists and would steer the agent toward ~/.claude/skills. Two changes fix that:

1 · Harness system-prompt block

Add a section to buildLaniHarnessBlock() (the append on the claude_code preset):

Skills live in ~/.lani/skills/ — one folder per skill.
To create, edit, or delete a skill, call propose_skill_change —
never edit SKILL.md files with raw file tools. Changes are
reviewed by the user and apply to the next session.

2 · Lani-adapted skill-creator (a factory skill)

Ship an adapted skill-creator in the factory set. It keeps the original's writing guidance — skill anatomy, progressive disclosure, frontmatter, description style — and replaces the environment-specific parts:

  • Target directory is ~/.lani/skills/<slug>/, stated explicitly — not ~/.claude/skills.
  • Writing the skill goes through propose_skill_change, not raw Write.
  • The heavy eval / benchmark / packaging apparatus is trimmed — a Lani writer wants to capture a workflow, not run a benchmark harness.

Harness block = the always-on rule. Adapted skill-creator = the how-to the agent reads when actually authoring. Belt and suspenders.
