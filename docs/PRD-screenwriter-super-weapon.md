# PRD — Backlot: The AI Screenwriter Super-Weapon

**Status:** vision doc · captured 2026-05-05 · revisit before Phase E begins
**Owner:** Michael
**Audience:** future Backlot builders + this conversation's continuation

---

## Vision

Backlot is the desktop workspace where an AI-native filmmaker builds the **entire** visual narrative pipeline — from idea to shot list — with the AI agent as a creative collaborator at every layer.

It is **not** "a screenplay editor with AI bolted on." It is a structured project workspace where the agent understands the relationships between locations, characters, scenes, and shots, and can write/rewrite/expand any of them in their proper context — automatically respecting identity locks, world art-direction, and continuity.

**Competitive frame:**
- Final Draft + Scrivener handle text
- Midjourney + Nano Banana Pro + Runway + Seedance + Kling handle pixels
- *Nothing* handles the connective tissue — the structured project that knows "this character looks like X, in this location, in scene 3, shot 4 — generate a starting frame prompt that respects all of that"

That gap is Backlot.

## The Project Hierarchy

```
Project
│
├── Locations         — places (e.g. "European forest road", "Ferrari pit garage")
├── Characters        — people, vehicles, creatures, anything with identity continuity
├── World             — art direction, palette, tone, period, technology level
├── Script (master)   — the screenplay overview spanning all scenes
│
└── Scenes            — each scene has ONE master script
    │
    └── Shots         — each shot has prompts to generate image / video
        │
        └── Prompts   — actual generative AI prompts (start frame, continuation, refs)
```

Every entity is a markdown file in the project's git repository (extending the Fountain convention we already use). Every entity is independently **forkable** as its own Direction.

### Entity definitions

| Entity | What it is | Why it has its own surface |
|---|---|---|
| **Location** | A place that recurs in the project. Has reference images, art-direction notes, time-of-day variants, lighting setups. | Agent reads when writing scenes set there to keep details consistent. |
| **Character** | Person, vehicle, animal, creature. Has a "lock" file — canonical description every prompt referencing this character pastes verbatim. Has wardrobe sheets, expression sheets, voice/personality. | Identity continuity across hundreds of prompts. Forkable: rewrite a character's voice mid-project. |
| **World** | The project's art-direction bible. Palette, tone, era, lens choices, look philosophy. One per project. | Used by every prompt as a "world lock." |
| **Script (master)** | The top-level screenplay. What we have today as `screenplay.fountain`. | Reframed as ONE artifact among many — not the whole project anymore. |
| **Scene** | A unit of dramatic action set in one Location with one set of Characters. Has its own script (Fountain), its own master version, optional variant Directions. | Most common forking unit. "Try scene 4 a different way" is the daily question. |
| **Shot** | A single camera setup within a scene. Starting-frame prompt, continuation prompt, references, video prompt. | The unit that actually gets generated by an AI video model (Seedance, Kling). |

### Why this hierarchy is right for screenwriters using AI

The user's daily question: *"what should I write next?"*
At the AI-filmmaking level it becomes: *"what's the next prompt?"*

That prompt depends on:
- Which scene we're in
- Which characters are in it (their identity locks)
- What location (its lighting/mood)
- The world (its art direction)

**Today:** you re-paste these into every prompt by hand. Or you maintain `_kb/` and `lock.md` files in a markdown workspace and copy-paste manually. (This is exactly what the laniameda-hq AI Creatorship workspace already does in markdown form.)

**Backlot:** the agent automatically composes them when you ask for a shot prompt. The structured project is the agent's working memory.

## Forking semantics across the hierarchy

Forking applies at **every** level. Not just at the project root.

| Level | What "fork" means | Frequency |
|---|---|---|
| Project | A whole alternate take on the project. | Rare |
| Script (master) | A and B versions of the overall arc. | Common |
| **Scene** | **Try a different version of just scene 4.** | **Most common** |
| Shot | Try a different camera angle for one beat. | Common |
| Character | "What if Alex was 60 instead of 30?" | Occasional |
| Location | "What if it was night instead of dawn?" | Occasional |

Every fork is its own git branch + worktree, exactly like today's Direction model.

The Story Canvas (Phase 5 in the current roadmap) expands from "tree of scripts" to "tree of any forkable entity, layered by hierarchy level."

## The Prompts View

A new view mode alongside Editor / Preview / Split / History: **Prompts**.

When you select a Shot in the project tree, the Prompts view shows:

- **Starting-frame prompt** (Nano Banana Pro / Midjourney) — text + composer
- **Continuation prompt** (Seedance / Kling) — text + composer
- **References gallery** — reference images attached, lock blocks pinned
- **Generated outputs** — links to AI-generated images/videos saved to the project
- **Iteration log** — every prompt variation tried, with resulting output

Per-prompt actions:
- **Edit** → autosaves to disk
- **Compose** → agent generates the prompt by combining: scene description + character locks + location + world locks
- **Iterate** → agent suggests N variations
- **Send to model** → (later) wires directly into Nano Banana / Seedance / Kling APIs

The Prompts view is what makes Backlot the **super-weapon** — it eliminates the manual step of assembling locks every time, which is currently the single biggest source of friction in AI filmmaking workflows.

## What we already have (re-framed against this vision)

| Current Backlot feature | New role under this vision |
|---|---|
| Screenplay artifact + Edit/Write/diff/accept | Edit surface for Script (master) **and** Scene scripts. Same flow, applied at multiple levels. |
| Fountain outline parser | Basis for auto-detecting scenes from the master script. |
| Per-line / per-hunk diff review | Same review UX applies to character locks, location notes, scene scripts. |
| Forking phase 1+2 (just shipped) | Generic "fork a chat tied to a worktree" — applies to scene-level forks too once we add the Project Hierarchy. |
| Direction sidetabs (phase 3, building) | Become "Direction tabs *for the currently-selected entity*." Switch entity → tabs reflect that entity's variants. |
| Story Canvas (phase 5, planned) | Becomes the **hierarchy canvas** — pan through all locations, characters, scenes, shots, see relationships, click any node to enter that entity's workspace. |

**Key insight:** nothing currently shipped is wasted. Forking + sidetabs + canvas all apply at multiple levels. The hierarchy adds **what the active "thing" is**, not how forking works on it.

## Open design questions (defer to Phase E kickoff)

1. **DB tables vs filesystem?** Today we use filesystem + git for `screenplay.fountain`. Should locations / characters / scenes also live as markdown files (e.g., `characters/lana.md`, `scenes/03-warehouse.fountain`)? Or as DB rows?
   - **Recommendation:** filesystem convention with a thin DB index for fast tree rendering and cross-references. Keeps git-friendly + forkable + matches the existing AI Creatorship OS pattern in `laniameda-hq/AI Creatorship/_os/`.

2. **How does the master script know its scenes?** Markdown anchors? Folder convention? Relations table?
   - **Recommendation:** folder convention. `scenes/01-opening.fountain`, `scenes/02-warehouse.fountain`. Master script is auto-stitched from them, or hand-curated to override order.

3. **Fork unit per level.** Project = whole git repo branch. Scene = could be a folder branch (whole `scenes/03-warehouse/`) or filename suffix like `scenes/03-warehouse.alex-pov.fountain`. Trade-offs.
   - **Recommendation:** decide during Phase E1 design — don't lock in early.

4. **Generated assets — local or external?**
   - **Recommendation:** pointers to laniameda.gallery + local cache for offline browsing. Keeps the worktree small + leverages the gallery skill that already exists.

5. **Identity-lock injection mechanism.**
   - **Recommendation:** both. Auto-prepend to prompt-composition flows AND `@mention` syntax for explicit references. Auto-inject is the killer feature; mention is for transparency / control.

## Sequencing — what this PRD does NOT change

The current Backlot work-in-flight (forking sidetabs phase 3, breadcrumb phase 4, canvas phase 5) operates at the simpler "one screenplay per project" model.

**Do not pause it.** Ship those phases. Build the muscle memory of forking + the canvas with the simple data model first. Every piece of UI shipped applies to the multi-entity world too.

Phase E (project hierarchy) starts as a **separate track** after Phase 5 lands. Existing single-screenplay projects migrate to "projects with one default scene" automatically — backwards compatible.

---

## Why this matters strategically

If Backlot ships only as "AI-assisted screenplay editor," it competes with Final Draft + ChatGPT-as-plugin. That's a crowded mid-market.

If Backlot ships as "the structured AI-filmmaking project workspace where every entity is composable and forkable," it has no direct competitor. It becomes the tool an AI creator opens **first** when starting a new project — the spine of their entire workflow.

The forking + canvas + chat-history-as-state we're building is the **scaffolding** for that. The hierarchy described in this PRD is the **content** that lives on the scaffolding.
