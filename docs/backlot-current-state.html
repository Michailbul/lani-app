Starting a project — copy, not worktree

works

When you import a folder or create a project, Backlot copies the source into its own space. It never works against your original folder in place.

Where projects live

Every project is a folder under ~/.backlot/projects/<slug>/ with its own fresh git repo. The copy is deliberate: Backlot's git history stays independent and the original is never touched.

Worktrees / forking — removed

The old Directions feature forked a project into extra git worktrees under ~/.backlot/worktrees/. That is gone. One project = one folder. No new worktrees are created; ~/.backlot/worktrees/ only holds stale leftovers from before the removal.

02

The working directory (cwd)

works

The agent — Claude or Codex — runs with its cwd set to the project folder itself.

// chats.ts — createChat
db.update(chats).set({ worktreePath: project.path })
// → the agent process spawns with:
cwd = ~/.backlot/projects/<slug>/

The DB column is still called worktreePath for legacy reasons, but it now always equals the project path. Every chat in a project shares the same cwd — there is no per-chat isolation anymore.

03

Skills — what Claude actually has

broken

Claude is given almost no skills today. Not because they aren't installed — because of one option.

// claude.ts — query() options
settingSources: ["project"]   ← "user" is missing

The SDK only discovers skills for the setting sources you enable. With just ["project"]:

  • User skills — ~/.claude/skills/ (~234 of them) → not loaded. "user" is not an enabled source.
  • Project skills — <project>/.claude/skills/ → loaded, but most projects have no .claude/ folder at all.
  • Plugin / CLI-bundled skills → loaded. This is the lone keybindings-help the agent reported having.

So in practice Claude sees a handful of built-in skills and nothing of yours. Asking it for screenwriter or nano-banana-pro returns "no such skill."

the fixRestore settingSources: ["project", "user"] — exactly what 1code uses. One line.

The Skills settings tab

The tab writes a filter to ~/.backlot/skills-filter.json — but resolveActiveSkillNames(), the function meant to apply it, has zero callers. Nothing reads the filter when a session starts. The tab currently controls nothing.

04

Skills — what Codex has

none

codex.ts has no skill wiring whatsoever — no skill discovery, no SKILL.md scan, no settingSources, no symlink.

Agent Skills are an Anthropic feature of the Claude Agent SDK. Codex has no equivalent primitive. A Codex session in Backlot runs on the harness alone. (1code is the same — its Codex path also has zero skills.)

06

CLAUDE.md — where memory comes from

partial

Project memory

Because settingSources includes "project", the SDK loads the project's CLAUDE.md — ~/.backlot/projects/<slug>/CLAUDE.md. That is the per-project memory the agent reads every turn.

Your global CLAUDE.md

Your personal ~/.claude/CLAUDE.md (the Laniameda studio doc) is not loaded — "user" is off, and Backlot deliberately never symlinks CLAUDE.md into the config dir. That isolation is intentional and should survive the skills fix: when "user" is restored for skills, CLAUDE.md still must not be symlinked, so user skills load without the studio doc bleeding in.

The harness

The Backlot harness (harness-prompt.ts) is separate from any CLAUDE.md. It is appended to the system prompt via systemPrompt.append for Claude, and passed as system for Codex. Same text for every project.

08

Subagents / agents

partial

Claude

Subagents are passed explicitly to the SDK through options.agents:

// claude.ts
agentsOption = { ...builtinAgents, ...mentionedAgents }
// builtinAgents  — Backlot's own subagents (e.g. director-verifier),
//                  minus any disabled in Settings
// mentionedAgents — agents the user @-mentions in the prompt
options.agents = agentsOption   // only when non-empty
  • Built-in subagents — always registered (unless disabled in Settings).
  • @-mentioned agents — resolved on demand from the mention.
  • Project agents (<cwd>/.claude/agents) — load via settingSources: ["project"].
  • User agents (~/.claude/agents) — symlinked into the config dir, but dead for the same reason as skills: "user" is off.

The SDK's per-subagent definition also has a skills: string[] field — a subagent can preload specific skills. Backlot does not use it yet.

Codex

No subagents. Codex has no equivalent.
